# JOGA Visualizer — Coaching Intelligence System Implementation Plan (Merged)

This document is the comprehensive implementation plan for transforming JOGA Visualizer from a dashboard/charting app into a proactive coaching intelligence system. It integrates the main architecture plan with detailed Context Cache implementation steps. Written to be interpretable by both humans and AI coding assistants (Cursor, Claude, etc.).

## Architecture Overview

The existing app is a React 18 + TypeScript frontend with a Vite bundler, an Express.js + TypeScript backend, PostgreSQL via Kysely, Recharts for charts, and Google Gemini 2.5 Flash for AI. Auth is session-based. The app currently has premade charts, a custom chart builder, and a chatbot.

The transformation adds five layers on top of this foundation, built in dependency order:

```
Layer 1: Insights Engine (statistical detection — the analytical backbone) ✅ COMPLETED
Layer 2: Training Log (coach input — closes the feedback loop) ✅ COMPLETED
Layer 3: Framework Integration + Gemini Context Caching (knowledge layer) ✅ COMPLETED
Layer 4: Recommendation Engine (AI-powered training suggestions) ⏳ PENDING
Layer 5: Workflow Views + Conversational Analysis (frontend reorganization) ⏳ PENDING
```

Nothing below requires a framework change, a rewrite, or new infrastructure. It's all additive: new database tables, new backend services, new API routes, and new frontend views alongside the existing ones.

### 1.2 Insights Service

Create `backend/src/services/insightsService.ts`.

This is the analytical backbone. It runs deterministic statistical checks — no AI needed for detection, only for narration.

**Detection algorithms to implement:**

#### A. Anomaly Detection (single-game outliers)
For each numeric stat for a given team, compute the season mean and standard deviation. Flag any single-game value with |z-score| > 1.5.

```
Input: team's matches for the season, the latest match
Output: list of { stat_name, value, season_avg, season_stddev, z_score }
Filter: only include where abs(z_score) > 1.5
Severity: map abs(z_score) to 0-1 range (1.5 → 0.3, 2.0 → 0.5, 2.5 → 0.7, 3.0+ → 1.0)
```

#### B. Trend Detection (multi-game directional changes)
Compute 3-game and 5-game rolling averages for key stats. Compare the rolling average to the season average. Flag when the rolling average diverges by more than 1 standard deviation.

```
Input: team's last N matches ordered by date
Key stats to track: possession, pass strings 4+, pass strings 6+, SPI, conversion rate,
                     attempts for, attempts against, xG (if available), pass share, PPM
Output: list of { stat_name, rolling_avg_3, rolling_avg_5, season_avg, direction: 'improving'|'declining', games_in_trend }
Severity: based on magnitude of divergence and number of consecutive games in the trend direction
```

#### C. Half-Split Analysis (1st vs 2nd half patterns)
Compute the delta between 1st half and 2nd half for each stat, per game. Track the rolling average of those deltas. Flag when a team consistently performs worse (or better) in one half.

```
Input: team's matches with half-split data
Stats to split: goals, shots/attempts, passes, possession (if half-split available), corners, free kicks
Output: list of { stat_name, avg_1st_half, avg_2nd_half, delta, delta_trend_direction, consecutive_games_with_drop }
Severity: based on size of delta relative to the stat's scale and consistency (number of consecutive games)
```

This is unique to JOGA's dataset. Most youth analytics tools don't have half-split data. Lean into it.

#### D. Correlation Analysis (stat relationships)
With 30 games/season per team, compute Pearson correlations between key stat pairs and outcomes (win/loss/draw coded as 3/1/0 or goal differential).

```
Key correlations to check:
  - pass strings 4+ vs result
  - pass strings 6+ vs result
  - SPI vs result
  - possession vs result
  - conversion rate vs result
  - 2nd half possession delta vs result
  - attempts for vs xG (if both available)
  - PPM vs possession
Output: list of { stat_a, stat_b, pearson_r, p_value_approx, sample_size }
Filter: only include where abs(r) > 0.4 and sample_size >= 8
Severity: based on abs(r) strength (0.4 → 0.3, 0.6 → 0.5, 0.8+ → 0.9)
```

Run correlation analysis only when a team has 8+ games (not after every match — run weekly or when game count crosses thresholds: 8, 15, 25).

#### E. Cross-Team Benchmarking
Compare a team's key stats against the club-wide averages (across all 10 teams). Normalize by age group if possible (use `teams.age_group`).

```
Input: all teams' season averages for key stats
Output: list of { team_id, stat_name, team_avg, club_avg, percentile_rank }
Filter: only surface where a team is in the top or bottom 20% of the club
Severity: based on distance from club average
```

Run after every match weekend (batch job), not per-match.

#### 3.1.2 Create Cache Manager Service

Create `backend/src/services/aiCacheManager.ts`:

```typescript
import { db } from '../db/database';
import crypto from 'crypto';

interface CacheMetadata {
  cacheId: string;
  expiresAt: Date;
  dataHash?: string;
}

/**
 * Get stored cache from database
 */
export async function getStoredCache(teamId: number, cacheType: string = 'combined'): Promise<CacheMetadata | null> {
  const cache = await db
    .selectFrom('ai_cache')
    .selectAll()
    .where('team_id', '=', teamId)
    .where('cache_type', '=', cacheType)
    .orderBy('created_at', 'desc')
    .limit(1)
    .executeTakeFirst();

  if (!cache) return null;

  return {
    cacheId: cache.cache_id,
    expiresAt: new Date(cache.expires_at),
    dataHash: cache.data_hash ?? undefined,
  };
}

/**
 * Check if cache is still valid
 */
export function isCacheValid(
  cache: CacheMetadata,
  currentDataHash: string
): boolean {
  // Check expiration
  if (new Date() >= cache.expiresAt) {
    return false; // Expired
  }

  // Check if data changed (if hash provided)
  if (cache.dataHash && cache.dataHash !== currentDataHash) {
    return false; // Data changed
  }

  return true; // Cache is valid
}

/**
 * Store cache metadata in database
 */
export async function storeCacheMetadata(metadata: {
  teamId: number;
  cacheType: string;
  cacheId: string;
  expiresAt: Date;
  dataHash?: string;
}): Promise<void> {
  // Delete old cache for this team/type
  await db
    .deleteFrom('ai_cache')
    .where('team_id', '=', metadata.teamId)
    .where('cache_type', '=', metadata.cacheType)
    .execute();

  // Insert new cache
  await db
    .insertInto('ai_cache')
    .values({
      team_id: metadata.teamId,
      cache_type: metadata.cacheType,
      cache_id: metadata.cacheId,
      expires_at: metadata.expiresAt.toISOString(),
      data_hash: metadata.dataHash ?? null,
    })
    .execute();
}

/**
 * Hash match data to detect changes
 */
export function hashMatchData(matchData: any[]): string {
  const crypto = require('crypto');
  // Create hash of key match data fields
  const dataString = JSON.stringify(
    matchData.map(m => ({
      id: m.id,
      date: m.match_date || m.date,
      opponent: m.opponent_name || m.opponent,
      // Include key metrics that would affect recommendations
      goalsFor: m.goalsFor,
      goalsAgainst: m.goalsAgainst,
      possession: m.possession,
      // ... other key fields
    }))
  );
  return crypto.createHash('sha256').update(dataString).digest('hex');
}

/**
 * Invalidate cache for a team (call when data changes)
 */
export async function invalidateTeamCache(teamId: number, cacheType: string = 'combined'): Promise<void> {
  await db
    .deleteFrom('ai_cache')
    .where('team_id', '=', teamId)
    .where('cache_type', '=', cacheType)
    .execute();
}
```

#### 3.4.1 Update Gemini Service

Update `backend/src/services/aiService.ts`:

```typescript
import { GoogleGenerativeAI } from '@google/generative-ai';
import { GoogleAICacheManager } from '@google/generative-ai/server';
import { getStoredCache, isCacheValid, storeCacheMetadata, hashMatchData, invalidateTeamCache } from './aiCacheManager';
import { buildTeamContext, TeamContextData } from './teamContextBuilder';
import { loadUSSFrameworks, loadClubPhilosophy } from './frameworkLoader';

const API_KEY = process.env.GEMINI_API_KEY!;
const genAI = new GoogleGenerativeAI(API_KEY);
const cacheManager = new GoogleAICacheManager(API_KEY);

/**
 * Get or create cache - ONLY called when AI feature is used (lazy creation)
 */
async function getOrCreateCache(teamId: number, contextData: TeamContextData): Promise<string> {
  // 1. Check database for existing cache
  const existingCache = await getStoredCache(teamId, 'combined');
  const currentDataHash = hashMatchData(contextData.matches);
  
  // 2. Check if cache is still valid
  if (existingCache && isCacheValid(existingCache, currentDataHash)) {
    // Cache valid - return it (NO API CALL, NO COST)
    return existingCache.cacheId;
  }
  
  // 3. Cache expired or missing - create new one (ONLY NOW)
  console.log('🔄 Cache expired or missing - creating new cache for team', teamId);
  
  // Load frameworks
  const [usSoccerFrameworks, clubPhilosophy] = await Promise.all([
    loadUSSFrameworks(),
    loadClubPhilosophy(),
  ]);
  
  // Build the system instruction from framework + philosophy + coaching rules
  const systemInstruction = buildTeamSystemInstruction(contextData, usSoccerFrameworks, clubPhilosophy);
  
  // Build the contents array — this is the "heavy" context that gets cached
  const contents = [
    {
      role: 'user' as const,
      parts: [{ text: contextData.formattedMatchData }],
    },
    {
      role: 'model' as const,
      parts: [{ text: 'I have received and understood the team data, development framework, insights, and training history. I am ready to answer questions about this team.' }],
    },
  ];
  
  // Create the cache on Gemini's server
  const cachedContent = await cacheManager.create({
    model: 'models/gemini-2.5-flash',
    systemInstruction: systemInstruction,
    contents: contents,
    ttlSeconds: 3600, // 1 hour
  });
  
  // Store cache metadata in database
  const expiresAt = new Date(Date.now() + 3600000); // 1 hour from now
  await storeCacheMetadata({
    teamId,
    cacheType: 'combined',
    cacheId: cachedContent.name!,
    expiresAt,
    dataHash: currentDataHash,
  });
  
  return cachedContent.name!;
}

/**
 * Build system instruction with frameworks and philosophy
 */
function buildTeamSystemInstruction(
  contextData: TeamContextData,
  usSoccerFrameworks: any,
  clubPhilosophy: any
): string {
  const { developmentStage, clubPhilosophy: club } = contextData;
  
  return `You are an AI assistant helping ${club.name || 'JOGA'} coaches analyze their match data and provide training recommendations.

US SOCCER DEVELOPMENT FRAMEWORK:
Development Stage: ${developmentStage.name} (${developmentStage.ageRange})
${developmentStage.overview}

Technical Priorities: ${developmentStage.technicalPriorities.join(', ')}
Tactical Priorities: ${developmentStage.tacticalPriorities.join(', ')}

CLUB PHILOSOPHY:
${club.playingIdentity}
Core Values: ${club.coreValues.join(', ')}
Non-negotiables: ${club.nonNegotiables.join(', ')}

All recommendations must align with BOTH:
1. US Soccer coaching standards (appropriate for ${developmentStage.ageRange})
2. ${club.name} club philosophy and playing style

When there is a conflict, prioritize club philosophy but explain the US Soccer standard.`;
}

/**
 * Chat using cached context — only the coach's message is sent as new tokens
 */
export async function chatWithCachedContext(
  teamId: number,
  message: string
): Promise<string> {
  // Build team context
  const contextData = await buildTeamContext(teamId, currentSeasonId);
  
  // Get or create cache (lazy creation)
  const cacheId = await getOrCreateCache(teamId, contextData);
  
  // Retrieve the full CachedContent object by name
  const cachedContent = await cacheManager.get(cacheId);
  
  // Create a model instance bound to this cache
  const model = genAI.getGenerativeModelFromCachedContent(cachedContent);
  
  // Generate — only the new message counts as non-cached tokens
  const result = await model.generateContent({
    contents: [{ role: 'user', parts: [{ text: message }] }],
  });
  
  return result.response.text();
}

/**
 * Invalidate a team's cache (call after match upload, training log entry, etc.)
 */
export async function invalidateTeamCacheForDataChange(teamId: number): Promise<void> {
  await invalidateTeamCache(teamId, 'combined');
}
```

**Key implementation notes:**
1. `cacheManager.create()` returns a `CachedContent` with a `name` field (server-side ID like `"cachedContents/abc123"`). Store this name in the database.
2. `genAI.getGenerativeModelFromCachedContent()` takes the full `CachedContent` object (retrieved via `cacheManager.get(name)`), not just the name string.
3. The `contents` array in the cache must follow the standard conversation format: `{ role: 'user' | 'model', parts: [{ text: string }] }`.
4. `systemInstruction` can be a plain string — the SDK handles conversion internally.
5. You CANNOT set `systemInstruction` again when calling `generateContent` on a model created from a cache. The system instruction is baked into the cache.
6. Call `invalidateTeamCacheForDataChange(teamId)` whenever match data is uploaded, training logs are added, or insights are regenerated for a team.

**SDK version note:** The installed `@google/generative-ai@^0.21.0` has the full caching API. The `@google/generative-ai` package was archived Dec 16, 2025, with replacement `@google/genai` (v1.40.0+). You do NOT need to migrate now — v0.21.0 works. Plan to migrate eventually. The conceptual structure is identical; only import paths and method signatures change.

**Minimum cache size:** Gemini requires at least 2,048 tokens in a cache. Your per-team context (~27K tokens) is well above this threshold.

### 4.2 Recommendation Service

Create `backend/src/services/recommendationService.ts`.

**Generation flow:**

```
1. Receive insight(s) for a team
2. Fetch team's training log (last 4-6 weeks) to know what's already been trained
3. Fetch team's development stage from framework
4. Build a recommendation prompt that includes:
   - The statistical finding (from the insight)
   - What the team has recently worked on (from training logs)
   - The age-appropriate guidelines (from the framework)
   - The club philosophy (from JOGA philosophy)
   - Whether to suggest a new focus or a progression of a previous one
5. Call Gemini (using the cached team context) with a specific recommendation prompt
6. Parse and store the recommendation
```

**Recommendation prompt template:**

```
Based on the following insight for {team_display_name} ({age_group}):

FINDING: {insight.title}
DETAIL: {insight.detail_json parsed into readable form}

RECENT TRAINING FOCUS (last 4 weeks):
{training_logs formatted as: date - focus tags - notes}

DEVELOPMENT STAGE: {developmentStage.name}
Key guidance for this age: {developmentStage.overview}

CLUB PHILOSOPHY: {clubPhilosophy.playingIdentity}
Non-negotiables: {clubPhilosophy.nonNegotiables}

Generate a training recommendation that:
1. Directly addresses the statistical finding
2. Is age-appropriate for {age_group} per the development framework
3. Aligns with JOGA's possession-based philosophy
4. Builds on (doesn't repeat) what was recently trained
5. Includes a specific exercise or session structure suggestion
6. Explains WHY this addresses the finding (connect the data to the training)

Format as JSON:
{
  "title": "short title (under 60 chars)",
  "recommendation_text": "full recommendation (2-4 paragraphs)",
  "focus_tags": ["tag_name_1", "tag_name_2"],
  "difficulty_progression": "introductory|reinforcement|progression|challenge"
}
```

**Progression logic:**

If the training log shows the coach already worked on a related focus:
- First time: `introductory` — new exercise/concept
- After 1 logged session on this topic: `reinforcement` — same concept, increased complexity
- After 2-3 logged sessions: `progression` — add constraints, game realism, pressure
- After 4+ logged sessions: `challenge` — full game scenario, match conditions

### 4.3 Trigger Mechanism

Recommendations are generated:
1. **Automatically** when a new insight with severity > 0.5 is created (post-match)
2. **On demand** when a coach requests a recommendation from the insight feed
3. **Weekly** as a batch job that looks at the week's insights and generates a "weekly training priorities" recommendation

## Key Technical Decisions and Notes

### Gemini Context Caching TTL Strategy
- Set TTL to 1 hour for match data cache (data changes infrequently)
- Invalidate cache explicitly when new match data is uploaded for a team
- Framework/philosophy content rarely changes — could be a longer TTL or a separate cache
- **Lazy creation**: Cache is only created when AI features are used, not on app load

### Data Volume Reality Check
- 10 teams × 30 games/year = 300 matches/season total
- Per-team: 30 data points is enough for z-scores and rolling averages, marginal for correlations (hence the 8-game minimum threshold)
- Context size per team: ~30 matches × ~500 tokens/match = ~15K tokens of match data. Plus framework (~5K), philosophy (~2K), insights (~3K), training logs (~2K) = ~27K tokens cached. Well within Gemini 2.5 Flash's context window.

### Cost Optimization
- **Cached tokens**: ~90% cheaper than regular input tokens
- **Lazy creation**: Only create cache when AI is actually used
- **Cache reuse**: Subsequent requests use cached tokens (cheap)
- **Smart invalidation**: Only recreate when data actually changes

### Existing Code to Preserve
- All 18+ chart components (`src/components/*Chart.tsx`) — these continue to work as-is and get embedded in new views
- `matchStatsService.ts` computation pipeline — the insights engine reads from its output, doesn't change it
- Custom chart builder — remains available for coaches who want it
- Auth middleware, user/team management — unchanged
- Google Sheets integration — unchanged
